# Repository Ontology

## Purpose

Evaluate whether a repository needs an ontology and, where justified, establish the smallest useful semantic model grounded in repository evidence.

The ontology must help people or software answer concrete questions, make decisions, validate state, retrieve information, or control agent behaviour. Do not create an ontology merely to document every type, file, class, table, or dependency in the repository.

This skill supports three modes:

- **Assess**: evaluate existing semantic assets and determine what is missing.
- **Establish**: create an initial repository ontology.
- **Evolve**: propose controlled changes to an existing ontology.

Always perform assessment before establishment or evolution.

## Core principles

### Prefer the minimum sufficient semantic model

Use the least formal representation that satisfies the repository's needs:

1. **Glossary or taxonomy**
   Use when the primary need is consistent terminology or categorisation.

2. **Typed concept model**
   Use when systems or agents need explicit entities, relationships, and constraints.

3. **RDF or JSON-LD model**
   Use when globally identifiable entities, graph queries, or interoperability are required.

4. **OWL ontology**
   Use when formal semantics, inference, or logical consistency checking are required.

5. **Operational constraints**
   Use SHACL or an equivalent validation mechanism when runtime graph data must conform to explicit conditions.

Do not adopt a more expressive representation because it appears more sophisticated.

### Ground every material concept

A repository identifier is not automatically a domain concept.

Do not infer canonical meaning solely from:

- file names;
- class names;
- database table names;
- API route names;
- directory layout;
- comments generated by an agent;
- one isolated implementation.

Use multiple sources where possible. Treat ambiguous interpretations as hypotheses requiring review.

### Keep different semantic layers distinct

Classify concepts into one or more of these layers:

- **Domain**: business concepts, rules, events, and capabilities.
- **Architecture**: systems, services, components, interfaces, and data stores.
- **Implementation**: packages, modules, files, symbols, schemas, and dependencies.
- **Delivery**: builds, tests, deployments, environments, and releases.
- **Governance**: ownership, policy, risk, approval, and authority.
- **Agent operation**: goals, actions, tools, preconditions, effects, and evidence.

Do not collapse a domain concept into the class or table currently implementing it.

### Separate schema, instances, constraints, and policy

Classify each material statement as one of:

- **Schema knowledge (TBox)**: classes, properties, hierarchies, and general restrictions.
- **Instance knowledge (ABox)**: facts about concrete repository artefacts, systems, people, or events.
- **Validation constraints**: conditions that graph or instance data must satisfy.
- **Operational policy**: mutable permissions, workflows, approvals, and organisational rules.

Do not generalise an observed repository instance into a universal ontology axiom unless authoritative evidence explicitly supports that generalisation. Do not treat a validation rule or mutable policy as an essential domain truth.

### Preserve provenance and uncertainty

For each material assertion, record:

- source;
- repository location;
- revision or commit when relevant;
- evidence type;
- confidence;
- status;
- reviewer, when confirmed.

Use these assertion statuses:

- `observed`: directly present in repository evidence;
- `inferred`: reasoned from one or more observations;
- `confirmed`: accepted by an authorised human or authoritative source;
- `disputed`: conflicting meanings or evidence exist;
- `deprecated`: retained for compatibility but no longer preferred.

Never present an inferred assertion as confirmed.

## Inputs

Use the evidence available in the repository, including:

- README and contributor documentation;
- architectural decision records;
- diagrams and architecture documentation;
- domain documentation and glossaries;
- source code types, interfaces, and public symbols;
- API specifications and message schemas;
- event definitions;
- database schemas and migrations;
- configuration and deployment definitions;
- tests, fixtures, and acceptance scenarios;
- package and module metadata;
- ownership and governance files;
- existing RDF, OWL, JSON-LD, taxonomy, or knowledge-graph assets;
- issue, pull-request, and version-control history when available and materially useful.

Prefer authoritative and maintained sources over generated, duplicated, or historical documentation.

## Procedure

### 1. Establish the purpose

Identify the decision, workflow, or capability the ontology is intended to support.

Examples include:

- improving agent retrieval;
- mapping business capabilities to code;
- selecting tools or repository areas;
- tracing requirements to implementations and tests;
- understanding service and event relationships;
- enforcing architectural constraints;
- supporting impact analysis;
- sharing semantics across repositories or agents;
- recording provenance and confidence;
- validating whether an agent task is complete.

State explicitly who or what will consume and maintain the ontology.

If no concrete consumer or use case can be identified, recommend against establishing a formal ontology.

### 2. Inspect existing semantic assets

Search for:

- glossaries;
- taxonomies;
- architecture models;
- domain models;
- schemas;
- type systems;
- knowledge graphs;
- metadata vocabularies;
- policy definitions;
- generated documentation;
- existing ontology files.

Determine whether the repository already has an implicit or partial ontology distributed across several artefacts.

Do not assume that the absence of RDF or OWL means no ontology exists. Conversely, do not assume that the presence of a graph database or dependency graph constitutes an ontology.

### 3. Define competency questions

Create questions that the ontology must answer.

Each competency question must identify:

- the intended consumer;
- why the answer matters;
- the required concepts and relationships;
- an example answer;
- the repository evidence expected to support it;
- an acceptance test.

Example questions:

- Which service implements a given business capability?
- Which component owns a particular domain concept?
- Which events can change the state of an aggregate?
- Which APIs expose a particular data object?
- Which tests validate a requirement or invariant?
- Which components would be affected by changing an interface?
- Which repository source is authoritative for a concept?
- Which actions may an agent perform on a given artefact?
- Which approval or evidence is required before an action?
- Which assertions are confirmed, inferred, or stale?

Reject competency questions that are too broad to test.

### 4. Decide whether an ontology is warranted

Give one of these verdicts:

- `not-needed`
- `glossary-sufficient`
- `taxonomy-sufficient`
- `typed-concept-model`
- `semantic-graph`
- `formal-ontology`
- `operational-ontology`

Base the verdict on the competency questions, not on repository size or technical novelty.

Document why simpler alternatives such as conventional documentation, JSON Schema, OpenAPI, static analysis, or a dependency graph would not be sufficient.

### 5. Identify reusable concepts

Before defining new terms, inspect applicable existing standards and vocabularies.

Potential reuse areas include:

- software-project metadata;
- source-code metadata;
- package and dependency information;
- licences and software bills of materials;
- provenance;
- organisational ownership;
- time and versioning;
- security and vulnerability information.

For every external term, decide whether to:

- reuse directly;
- specialise;
- map as equivalent;
- map as broader or narrower;
- reject as unsuitable.

Do not import an entire ontology when only a small subset is required.

### 6. Build the evidence-backed term inventory

Extract candidate terms from repository evidence.

For each term record:

```yaml
term: Policy
preferred_label: Policy
definition: A contract defining cover, parties, limits, and validity.
layer: domain
status: inferred
evidence:
  - path: src/domain/Policy.cs
    kind: implementation
  - path: docs/domain-model.md
    kind: documentation
synonyms:
  - InsurancePolicy
conflicts:
  - The billing subsystem uses "policy" to mean a retry policy.
```

Separate:

- preferred terms;
- synonyms;
- homonyms;
- deprecated terms;
- implementation-specific names;
- unresolved terminology conflicts.

Definitions must distinguish the concept from adjacent concepts. Avoid circular definitions.

### 7. Construct the conceptual model

Model only the concepts and relationships needed by the competency questions.

For each concept define:

- identifier;
- preferred label;
- definition;
- semantic layer;
- broader concept where applicable;
- required and optional properties;
- identity criteria;
- lifecycle or state where relevant;
- provenance requirements;
- examples and counterexamples.

For each relationship define:

- identifier;
- label;
- definition;
- source concept;
- target concept;
- direction;
- cardinality where justified;
- inverse relationship where useful;
- temporal characteristics;
- supporting evidence;
- examples and counterexamples.

Prefer precise relationships such as:

- `implements`
- `owns`
- `publishes`
- `consumes`
- `dependsOn`
- `validates`
- `governedBy`
- `authorisedBy`
- `derivedFrom`
- `supersedes`

Avoid vague relationships such as `relatedTo` unless no more specific meaning is justified.

### 8. Generate candidate formal axioms

Perform this step only when the verdict requires a formal ontology.

Before generating axioms, declare:

- the target ontology language and serialisation;
- the permitted axiom and expression types;
- naming and identifier conventions;
- the expected ontology profile or expressivity boundary;
- whether the task concerns schema knowledge, instance knowledge, or both.

Permit only the constructs required by the competency questions. These may include:

- class, property, and individual declarations;
- class assertions and subclass relationships;
- disjointness and complements;
- object-property assertions;
- property domains and ranges;
- symmetric and asymmetric property characteristics;
- property hierarchies;
- existential, universal, and cardinality restrictions.

Do not introduce a construct merely because the target language supports it.

Translate evidence into candidate axioms using a constrained, structured output format. For every candidate record:

```yaml
candidate_id: AX-001
knowledge_kind: schema # schema | instance
interpretation: Every service that publishes an event is an event publisher.
formal_expression: SubClassOf(ObjectIntersectionOf(Service ObjectSomeValuesFrom(publishes Event)) EventPublisher)
evidence:
  - path: docs/eventing.md
    location: section 3
status: proposed
expected_inferences:
  - A service publishing an Event is classified as EventPublisher.
potential_unintended_inferences: []
confidence: medium
```

Verify that every referenced entity already exists or is intentionally declared as a new, evidence-backed candidate. Do not introduce entities, restrictions, cardinalities, inverses, disjointness, or property characteristics without supporting evidence.

Use an explicit grammar or schema, a small set of reviewed examples covering the permitted axiom families, constrained or validated structured output, and low-variance generation settings where available. Separate generation from critique and validation. Fall back to human-authored formalisation when ambiguity remains.

Candidate axioms are proposals. They must not directly modify the canonical ontology. Prefer constrained prompting and validation before fine-tuning. Consider fine-tuning only when the target formalism is stable, a sufficiently large reviewed corpus exists, demand is repeated, and measured evidence shows that constrained prompting is inadequate.

### 9. Model constraints separately from definitions

Distinguish:

- concept definitions;
- structural constraints;
- business invariants;
- architectural policies;
- agent permissions;
- workflow preconditions;
- inferred rules.

Example:

```yaml
action: MergePullRequest
preconditions:
  - required_checks_passed
  - required_approvals_present
permission:
  required: repository.merge
effect:
  - pull_request.status becomes merged
risk:
  level: medium
reversible: false
```

Do not encode mutable organisational policy as an essential truth about the domain.

### 10. Evaluate agentic suitability

When the ontology will support agents, check whether it represents:

- goals and completion conditions;
- actions and tool capabilities;
- action inputs and outputs;
- preconditions and effects;
- permissions and authority;
- risk and reversibility;
- evidence and provenance;
- source authority;
- confidence and uncertainty;
- time-dependent state;
- stale or superseded assertions;
- human approvals and verdicts.

An agent-facing ontology must help constrain or validate behaviour, not merely supply additional prompt context.

### 11. Work incrementally on large repositories

When the relevant evidence cannot be reviewed in one bounded context:

- partition evidence by domain boundary, subsystem, semantic layer, or competency question;
- extract candidate terms independently;
- reconcile identifiers and meanings before generating cross-boundary relationships;
- construct the high-level hierarchy before adding detailed relationships and restrictions;
- maintain an unresolved-conflict register across batches;
- rerun global consistency and competency-question validation after reconciliation.

Do not merge independently generated fragments without checking for duplicated concepts, conflicting definitions, incompatible identity criteria, and accidental cross-layer equivalence.

### 12. Validate against evidence and examples

Validation must include:

#### Competency-question coverage

For every competency question:

- execute or simulate the required query;
- show the expected answer;
- identify missing concepts or relationships;
- record whether free-text interpretation was still required.

#### Positive examples

Create representative valid instances.

#### Negative examples

Create examples that must be rejected or must not produce the same inference.

#### Consistency checks

Check for:

- contradictory definitions;
- incompatible domains or ranges;
- impossible cardinalities;
- circular definitions;
- unintended equivalences;
- orphaned concepts;
- duplicate relationships;
- terms without evidence;
- implementation details incorrectly promoted to domain concepts.

#### Formal ontology validation

When formal axioms are generated or changed:

1. Parse the generated serialisation.
2. Load it into a temporary ontology or isolated graph.
3. Verify that referenced entities are declared or intentionally external.
4. Check logical consistency and class satisfiability with an appropriate reasoner.
5. Detect duplicate, redundant, and semantically equivalent axioms.
6. Compare every axiom against its cited evidence and natural-language interpretation.
7. Test the intended competency-question inferences.
8. Inspect unintended inferred relationships and classifications.
9. Produce an ontology diff for human review.
10. Merge only accepted axioms and preserve rejected candidates with their verdicts.

Reasoner success proves only that the ontology is mechanically consistent under its semantics. It does not prove that the model expresses the intended business or architectural meaning. Keep mechanical validation and domain review as separate gates.

#### Candidate-generation evaluation

For an LLM-assisted formalisation pipeline, measure:

- syntactic validity rate;
- logical consistency and satisfiability rate;
- unsupported-axiom rate;
- missing-required-axiom rate;
- unnecessary-entity rate;
- semantic equivalence to reviewed reference examples;
- competency-question pass rate;
- repeated-run agreement;
- human acceptance, correction, and rejection rates.

Do not use token-level or string-exact accuracy as the sole quality measure. Equivalent axioms may have different serialisations, while syntactically valid axioms may encode materially different meanings.

#### Repository traceability

Verify that material concepts can be traced to current repository evidence.

#### Change resilience

Consider whether routine refactoring would incorrectly change domain identity or invalidate the ontology.

### 13. Evaluate ontology maturity

Assess each dimension using:

- `absent`
- `implicit`
- `defined`
- `operational`

Dimensions:

1. Purpose and scope
2. Competency-question coverage
3. Conceptual clarity
4. Terminology consistency
5. Evidence traceability
6. Relationship precision
7. Constraint coverage
8. Reuse and interoperability
9. Provenance and temporal modelling
10. Agent integration
11. Validation and automated checks
12. Ownership and change governance

Do not hide serious gaps behind a composite numerical score.

### 14. Apply human judgement gates

Human review is required before:

- accepting canonical business definitions;
- resolving disputed terminology;
- declaring two concepts equivalent;
- accepting generated formal axioms into the canonical ontology;
- changing identifiers used by other systems;
- importing a broad external ontology;
- introducing business or compliance rules;
- granting an agent new authority;
- publishing a breaking ontology version.

Present the evidence, alternatives, and consequences. Do not make the decision implicitly.

### 15. Establish repository artefacts

Adapt filenames to repository conventions, but prefer a structure similar to:

```text
ontology/
├── README.md
├── scope.md
├── competency-questions.md
├── glossary.md
├── concept-model.yaml
├── mappings.md
├── evidence.md
├── candidates/
├── examples/
├── decisions/
└── validation/
```

Create formal serialisation only when justified:

```text
ontology/
├── ontology.ttl
├── context.jsonld
└── shapes/
    └── repository-shapes.ttl
```

The human-readable model and decision record must remain understandable without requiring an ontology editor.

### 16. Define lifecycle and ownership

Document:

- ontology owner;
- domain reviewers;
- change proposal process;
- compatibility policy;
- versioning policy;
- validation commands;
- release process;
- deprecation process;
- triggers for reassessment.

Ontology changes should be reviewed like code. Agent-generated changes are proposals and must not silently redefine canonical semantics.

## Constraints

- Do not model every code symbol.
- Do not equate a code dependency graph with an ontology.
- Do not invent domain definitions from naming alone.
- Do not treat generated documentation as authoritative without verification.
- Do not silently resolve disputed terminology.
- Do not introduce RDF, OWL, or a graph database unless the competency questions justify them.
- Do not combine facts, policies, permissions, and hypotheses into one undifferentiated model.
- Do not generalise a repository instance into a universal axiom without authoritative evidence.
- Do not allow generated candidate axioms to update the canonical ontology directly.
- Do not treat syntactic validity or reasoner consistency as proof of semantic correctness.
- Do not let the ontology become a parallel source of truth disconnected from repository evidence.
- Do not allow an agent to approve its own canonical semantic changes.
- Prefer a small ontology that is tested and used over a comprehensive ontology that is not operational.
